     426              : # Extended help
     427              : 
     428              : ## Difference between `callback`, `affect!` and `affect_t!`
     429              : 
     430              : The difference between the `callback(iter)` and `affect!(iter)` functions is the time at
     431              : which they are called:
     432              : 
     433              : - the `affect!` function is called *before* performing Biot-Savart computations from the
     434              :   latest filament positions. In other words, the fields in `iter.quantities` are not
     435              :   synchronised with `iter.fs`, and it generally makes no sense to access them.
     436              :   Things like energy estimates will be incorrect if done in `affect!`. On the other hand,
     437              :   the `affect!` function **allows to modify `iter.fs`** before Biot–Savart computations are
     438              :   performed. In particular, to inject new filaments, call [`inject_filament!`](@ref) from
     439              :   within the `affect!` function.
     440              : 
     441              : - the `callback` function is called *after* performing Biot-Savart computations.
     442              :   This means that the fields in `iter.quantities` have been computed from the latest filament positions.
     443              :   However, one **must not modify `iter.fs`**, or otherwise the velocity `iter.vs` (which
     444              :   will be used at the next timestep) will no longer correspond to the filament positions.
     445              : 
     446              : In addition, one can also set an `affect_t!(iter, t)` function which can be used to modify
     447              : the state of the solver in a fine-grained manner, e.g. at each Runge–Kutta substep. Note
     448              : that this function must take an additional argument `t` (the current time) as a second argument.
     449              : This may be used, for instance, to apply a forcing which varies in time.
     450              : 
     451              : ## Forcing / energy injection methods
     452              : 
     453              : There are multiple implemented methods for affecting and injecting energy onto the vortex system.
     454              : These methods can represent different physical mechanisms (or in fact be quite unphysical), and some
     455              : of them can make more sense than others when simulating e.g. zero-temperature superfluids.
     456              : 
     457              : ### Adding an external velocity
     458              : 
     459              : One can set the **`external_velocity`** keyword argument to impose an external velocity field
     460              : ``\bm{v}_{\text{f}}(\bm{x}, t)``.
     461              : In this case, the total velocity at a point ``\bm{x}`` (which can be on a vortex filament) will be
     462              : 
     463              : ```math
     464              : \bm{v}(\bm{x}) = \bm{v}_{\text{BS}}(\bm{x}) + \bm{v}_{\text{f}}(\bm{x})
     465              : ```
     466              : 
     467              : where ``\bm{v}_{\text{BS}}`` is the velocity obtained from the Biot–Savart law.
     468              : 
     469              : This can be a way of applying an external forcing (and injecting energy) to the vortex system.
     470              : 
     471              : The external velocity should be given as a function, which should have the signature `v_ext(x⃗::Vec3, t::Real) -> Vec3`.
     472              : The function should return a `Vec3` with the velocity at the location `x⃗` and time `t`.
     473              : For example, to impose a constant sinusoidal forcing in the ``x`` direction which varies along ``z``,
     474              : the `external_velocity` keyword argument could look like:
     475              : 
     476              :     external_velocity = (x⃗, t) -> Vec3(0.1 * sin(2 * x⃗.z), 0, 0)
     477              : 
     478              : One usually wants the external velocity to be divergence-free, to preserve the
     479              : incompressibility of the flow.
     480              : 
     481              : #### Notes on energy estimation
     482              : 
     483              : When setting an external velocity ``\bm{v}_{\text{f}}(\bm{x}, t)``, one may also want to
     484              : impose an external streamfunction field ``\bm{ψ}_{\text{f}}(\bm{x}, t)`` via the **`external_streamfunction`**
     485              : keyword argument.
     486              : Such a field plays no role in the dynamics, but it allows to include the kinetic energy
     487              : associated to the external velocity field when calling [`Diagnostics.kinetic_energy_from_streamfunction`](@ref).
     488              : If provided, the external streamfunction should satisfy ``\bm{v}_{\text{f}} = \bm{∇} × \bm{ψ}_{\text{f}}``.
     489              : 
     490              : More precisely, when applying an external velocity field, the total kinetic energy is
     491              : 
     492              : ```math
     493              : \begin{align*}
     494              :     E &= \frac{1}{2V} ∫ |\bm{v}_{\text{BS}} + \bm{v}_{\text{f}}|^2 \, \mathrm{d}^3\bm{x}
     495              :     \\
     496              :     &= \frac{1}{2V} \left[
     497              :     Γ ∮_{\mathcal{C}} \left( \bm{ψ}_{\text{BS}} + 2 \bm{ψ}_{\text{f}} \right) ⋅ \mathrm{d}\bm{s}
     498              :     +
     499              :     ∫ |\bm{v}_{\text{f}}|^2 \, \mathrm{d}^3\bm{x}
     500              :     \right],
     501              : \end{align*}
     502              : ```
     503              : 
     504              : where ``\bm{ψ}_{\text{BS}}`` is the streamfunction obtained from the Biot-Savart law.
     505              : Accordingly, when passing `external_streamfunction`, the streamfunction values attached to
     506              : vortex filaments (in `iter.ψs`) will actually contain
     507              : ``\bm{ψ}_{\text{BS}} + 2 \bm{ψ}_{\text{f}}`` (note the factor 2) to enable the
     508              : computation of the full kinetic energy.
     509              : Moreover, calling [`Diagnostics.kinetic_energy_from_streamfunction`](@ref) with a
     510              : [`VortexFilamentSolver`](@ref) will automatically include an estimation of the kinetic
     511              : energy of the external velocity field (the volume integral above).
     512              : 
     513              : !!! note
     514              : 
     515              :     Whether this total kinetic energy is of any physical interest is a different question,
     516              :     so it can be reasonable to ignore the `external_streamfunction` parameter in order to
     517              :     obtain the energy associated to ``\bm{v}_{\text{BS}}`` only.
     518              : 
     519              : ### Adding a stretching velocity
     520              : 
     521              : One can set the **`stretching_velocity`** keyword argument to impose a stretching velocity
     522              : ``\bm{v}_\text{L}(ξ, t)`` on filament locations ``\bm{s}(ξ, t)``.
     523              : This velocity will be parallel (and generally opposite) to the curvature vector
     524              : ``\bm{s}'' = ρ \bm{N}``, where ``ρ`` is the curvature and ``\bm{N}`` the unit normal.
     525              : 
     526              : More precisely, the `stretching_velocity` parameter allows to set a filament velocity of the
     527              : form
     528              : 
     529              : ```math
     530              : \bm{v}_{\text{L}}(ξ) = -v_{\text{L}}[ρ(ξ)] \, \bm{N}(ξ)
     531              : ```
     532              : 
     533              : where ``v_{\text{L}}`` is a velocity magnitude which can optionally depend on the local
     534              : curvature value ``ρ``.
     535              : The `stretching_velocity` parameter should therefore be a function `v(ρ::Real) -> Real`.
     536              : 
     537              : Note that ``\bm{v} ⋅ \bm{s}''`` can be roughly interpreted as a local stretching rate
     538              : (this is actually true for the integrated quantity, see [`Diagnostics.stretching_rate`](@ref)).
     539              : Therefore, one may want ``v_{\text{L}}`` to be inversely proportional to the local
     540              : curvature ``ρ``, so that the local stretching rate is approximately independent of the
     541              : filament location ``ξ``.
     542              : To achieve this, one may set
     543              : 
     544              : ```julia
     545              : stretching_velocity = ρ -> min(γ / ρ, v_max)
     546              : ```
     547              : 
     548              : where `γ` is a constant (units ``T^{-1}``) setting the stretching magnitude, and `v_max` is
     549              : a maximum stretching velocity used to avoid very large velocities in low-curvature regions.
     550              : One could do something fancier and replace the `min` with a smooth regularisation such as
     551              : 
     552              : ```julia
     553              : stretching_velocity = ρ -> -expm1(-ρ / ρ₀) * (γ / ρ)  # note: expm1(x) = exp(x) - 1
     554              : ```
     555              : 
     556              : for some small curvature ``ρ₀``. The maximum allowed velocity will then be `vmax = γ / ρ₀`.
     557              : 
     558              : ### Forcing via a normal fluid
     559              : 
     560              : Alternatively to the above approaches, one may impose a "normal" fluid velocity field to
     561              : modify, via a mutual friction force, the velocity of the vortex filaments. To do this, one
     562              : first needs to construct a [`NormalFluidForcing`](@ref) (see that link for definitions and examples)
     563              : representing a normal fluid velocity field. Then, the obtained object should be passed as
     564              : the optional `forcing` argument.
     565              : 
     566              : In principle, this method can be combined with the previous ones. The normal fluid forcing
     567              : will be applied _after_ all the other forcing methods.
     568              : In other words, the vortex velocity ``\bm{v}_{\text{s}}`` used to compute the mutual
     569              : friction velocity includes all the other contributions (i.e. external and stretching velocities).
     570              : 
     571              : There is also a [`FourierBandForcing`](@ref) type which behaves similarly to
     572              : `NormalFluidForcing` but is more localised in scale space, as it uses a band-pass filtered
     573              : version of the superfluid velocity induced by the vortices.
     574              : 
     575              : ### Injecting filaments over time
     576              : 
     577              : Another way of injecting energy is simply by adding vortices to the simulation from time to
     578              : time. This can be achieved by using an `affect!` function. See [`inject_filament!`](@ref) for
     579              : some more details.
     580              : 
     581              : ## Energy minimisation mode
     582              : 
     583              : It is also possible to run the simulation in "energy minimisation" mode, which can be useful
     584              : for finding a minimal energy state from a given initial condition. In this mode, filaments
     585              : will be advected not using the Biot–Savart velocity ``\bm{v}_{\text{s}}``, but using the velocity
     586              : ``-\bm{s}' × \bm{v}_{\text{s}}`` where ``\bm{s}'`` is the local unit tangent vector.
     587              : 
     588              : This is because the functional derivative of the energy (per unit mass) with respect to the
     589              : vortex positions is:
     590              : 
     591              : ```julia
     592              : \frac{δE}{δ\bm{s}} = Γ \bm{s}' × \bm{v}_{\text{s}}
     593              : ```
     594              : 
     595              : In principle, the simulation should converge to a state that (locally) minimises the kinetic
     596              : energy. In some cases, e.g. when one starts with a single closed vortex, this is simply a
     597              : state where all vortices have disappeared.
     598              : 
     599              : Pass `mode = MinimalEnergy()` to enable this mode.
     600              : Forcing should be disabled to run in this mode.
